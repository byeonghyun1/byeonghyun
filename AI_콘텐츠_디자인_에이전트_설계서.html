1. 설계 원칙

이 시스템은 세 가지 원칙 위에 설계되었습니다.

유연성

입력 통로, 콘텐츠 유형, 브랜드가 추가되어도 엔진 코어는 수정하지 않는다.

확장성

새 기능은 기존 코드를 고치는 게 아니라, 새 모듈을 "꽂는" 방식으로 추가한다.

분리

입력(Input) · 엔진(Engine) · 출력(Output) 세 계층은 서로 직접 의존하지 않는다.

2. 전체 구조 (3계층)
입력 계층 (Input Layer)
Google Sheets / Slack / Web / CLI
▼ 표준 요청 (DesignRequest)
엔진 계층 (Engine Core)
분석 → 카피 → 스타일 → 템플릿 → 렌더링
▼ 표준 결과 (DesignResult)
출력 계층 (Output Layer)
파일저장 / Slack / Figma / Google Drive

각 계층은 '표준 요청 형식'으로만 소통합니다.
입력 통로가 바뀌어도, 콘텐츠 유형이 늘어나도, 엔진은 그대로입니다.

2
3. 표준 요청 형식 (DesignRequest)

어떤 입력 통로에서 오든, 엔진은 이 형식만 처리합니다.

기본 정보
content_type : 콘텐츠 유형 (카드뉴스, 배너, 포스터 등)
title : 제목
description : 설명 문구
mood : 분위기 (귀여운, 세련된, 활기찬 등)
브랜드 & 템플릿
brand : 브랜드명 (albamon, jobkorea, worxphere 등)
template_id : 사용할 템플릿 (비어있으면 자동 선택)
카피 모드 (NEW)
copy_mode : 카피 처리 방식 선택
  manual → 직접 입력 (기획팀 카피 그대로, AI 개입 없음)
  ai → AI가 처음부터 카피 생성
  refine → AI가 기존 카피를 다듬어서 개선
스타일 & 출력
style_overrides : 색상 등 직접 지정 (선택사항)
output_format : 출력 형식 (png, html, pdf)
output_size : 출력 크기 (가로 × 세로)
메타정보
source : 어디서 요청이 왔는지 (sheets, slack, web, cli)
request_id : 고유 요청 번호
requested_at : 요청 시각
3
4. 입력 어댑터 (변환 플러그)

각 입력 통로마다 '변환 플러그(어댑터)'가 하나씩 있습니다.
역할은 딱 하나: 외부 데이터를 DesignRequest 형식으로 바꾸는 것.

순서어댑터설명입력 형태
1CLI터미널에서 직접 입력커맨드라인
2Google Sheets시트에서 데이터 읽기행(row) 데이터
3Slack슬랙 메시지/명령어메시지 텍스트
4Web API웹사이트에서 요청JSON 데이터
핵심 규칙
  • 새 입력 통로 추가 = 어댑터 파일 하나만 추가
  • 어댑터는 엔진 내부를 절대 건드리지 않음
  • 기존 어댑터 수정 없이 새 통로 확장 가능
5. 엔진 코어 (조립 라인)

콘텐츠 유형에 상관없이 동일한 조립 라인을 실행합니다.

1단계: 요청 분석 자연어 해석, 빈 값 채우기, 유효성 검사
2단계: 카피 처리 manual = 건너뜀 / ai = 새로 생성 / refine = 다듬기
3단계: 스타일 결정 직접지정 > 브랜드 > 분위기 > 기본값 순서로 적용
4단계: 템플릿 매핑 콘텐츠 유형 + 브랜드 조합으로 템플릿 자동 선택
5단계: 렌더링 최종 이미지/파일 생성 (PNG, HTML, PDF)
4
6. 브랜드 스타일 시스템

브랜드별 디자인 규칙을 데이터(설정 파일)로 관리합니다.

새 브랜드 추가 = 폴더 하나 추가. 엔진 코드 수정 없음.

브랜드 폴더 구조
brands/
  albamon/    ← 브랜드 기본정보, 컬러, 폰트, 로고
  jobkorea/   ← 각 브랜드마다 같은 구조
  worxphere/ ← 새 브랜드는 폴더만 추가하면 됨
  superbrand/ ← 나중에 추가되는 브랜드도 동일
브랜드 프로필에 포함되는 것들
  • 컬러 시스템: 메인색, 보조색, 배경색, 글자색 등
  • 폰트 규칙: 제목/본문 폰트, 크기, 굵기
  • 에셋: 로고 이미지, 아이콘
  • 톤앤매너: 브랜드 성격, 타겟 정보
스타일 적용 우선순위 (위가 더 높음)
1순위직접 지정 (사용자가 색상 등을 수동 입력)
2순위브랜드 스타일 (brand.yaml에서 가져옴)
3순위분위기 프리셋 (귀여운, 세련된 등)
4순위시스템 기본값
7. 템플릿 시스템

콘텐츠 유형별로 HTML 템플릿을 관리합니다.

새 템플릿 = HTML 파일 + 목록에 등록. 엔진 코드 수정 없음.

templates/
  _base/       ← 공통 레이아웃 (뱃지, 구분선 등 재사용 부품)
  card_news/  ← 카드뉴스용 (기본형, 사진배경형 등)
  banner/     ← 배너용 (프로모션, 이벤트 등)
  poster/     ← 포스터용
  sns/        ← SNS 썸네일용

8. 확장 시나리오

시스템이 커져도 코드가 꼬이지 않는 구조입니다.

▸ 포스터 유형 추가
추가: 설정 파일 + 템플릿만 추가
→ 엔진 / 어댑터 / 브랜드 수정 없음
▸ Slack 입력 연동
추가: 슬랙 어댑터 파일 하나만 추가
→ 엔진 / 템플릿 / 브랜드 수정 없음
▸ 새 브랜드 추가
추가: 브랜드 폴더 하나만 추가
→ 엔진 / 어댑터 / 템플릿 수정 없음
▸ 웹사이트 연동
추가: 웹 어댑터 파일 하나만 추가
→ 엔진 코드 전체 수정 없음
9. 단계별 전환 계획

현재 카드뉴스 생성기 → 에이전트 시스템으로 전환합니다.

각 단계가 끝날 때마다 시스템은 정상 작동 상태를 유지합니다.

단계작업 내용기존 기능
1폴더 구조 정리 + 요청 양식 일반화그대로 작동
2브랜드 스타일을 설정 파일로 분리그대로 작동
3템플릿 시스템 구축템플릿 이관
4어댑터 계층 추가 (CLI 먼저)어댑터 사용
5새 콘텐츠 유형 추가 (배너 등)모두 작동
6Google Sheets 어댑터시트 입력 가능

10. 놓치기 쉬운 것들

시스템을 키워나갈 때 미리 챙겨야 할 포인트입니다.

▸ 에러 기록 시스템
지금은 에러가 나면 화면에 출력만 하고 사라집니다.
시스템이 커지면 '언제, 어디서, 왜 실패했는지' 기록이 필요합니다.
▸ 설정 한 곳 관리
API 키, 출력 경로 같은 설정이 코드 여기저기 흩어져 있습니다.
설정 파일 하나에서 모아 관리해야 합니다.
▸ 결과물 히스토리
같은 요청을 여러 번 돌리면 파일만 쌓입니다.
A/B 테스트를 하려면 '어떤 조건 → 어떤 결과'를 연결하는 기록이 필요합니다.
▸ 이미지 에셋 처리
배너나 포스터에는 배경 이미지, 상품 사진이 필요합니다.
이미지를 어디서 가져와서 어떻게 배치할지 구조가 필요합니다.
▸ 출력 통로 분리
지금은 파일 저장만 가능합니다.
Slack 전송, Google Drive 업로드 등 출력도 입력처럼 분리해야 합니다.
▸ 품질 피드백 루프
'시스템 자체 학습과 고도화' 목표를 위해 필요합니다.
결과물에 대한 평가(좋음/수정/재생성)를 기록하고 개선에 활용합니다.
