# 배너 자동화 빌더 PRD (초안)

작성일: 2026-04-28
작성자: 승연
상태: Draft v0.1

---

## 1. 배경

배너 제작은 같은 메시지를 사이즈·채널별로 변형해서 반복 출력하는 비중이 큰 작업이다. 디자이너가 사이즈마다 개별 레이아웃을 손으로 만들고, 카피·이미지만 바뀐 변형본을 다시 찍어내는 흐름이 자주 발생한다. 이 반복을 줄이고, 디자이너가 만든 템플릿을 자산으로 쌓아 추후 마케터·운영자도 활용할 수 있는 환경을 만든다.

\*1차 사용자는 디자이너로 좁힌다. 디자이너가 직접 운영하면서 템플릿 라이브러리·작업 히스토리를 축적한 후, 마케터/운영자에게 점진적으로 확장한다.

## 2. 목표

- 배너 1세트(여러 사이즈 묶음) 제작 소요 시간 단축
- 디자이너 운용 부담 감소 (반복 변형 작업 비중 축소)
- 추후 마케터·운영자가 디자이너 자산을 그대로 받아 운영 가능한 구조 확보

## 3. 비목표 (MVP 범위 밖)

- 광고 플랫폼(Meta/Google/Kakao Ads) 자동 업로드
- 외부 클라이언트용 공개 API
- 실시간 다수 동시 편집 (Figma·Notion 스타일)
- 브랜드 디자인 토큰 시스템

## 4. 사용자 / 시나리오

### 4-1. 주 사용자

- Phase 1 — 사내 디자이너
- Phase 2 (확장) — 사내 마케터, 운영자

### 4-2. 핵심 시나리오 (MVP)

1. 디자이너가 프로젝트를 생성하고 사용자 정의 사이즈를 여러 개 등록한다.
2. 슬롯 기반 + 재사용 컴포넌트 구조로 각 사이즈별 레이아웃을 개별 제작한다.
3. 빌더 UI에서 직접 텍스트 입력·이미지 업로드·AI 이미지 생성을 통해 소재를 채운다.
4. 자동 검증 (오탈자, 이미지 적합성, CTA 동작) 통과 후 승인 단계에 올린다.
5. 승인된 결과물을 일괄 다운로드한다.
6. 프로젝트는 버전·히스토리로 보존된다.

## 5. 배포 / 운영 방식

- 형태: 외부 공개 SaaS 지향. 단 MVP는 내부용으로 운영하며 외부 공개는 다음 단계에서 결정.
- 빌더 코드 → GitHub (버전 보존) → Vercel 배포 → 브라우저 빌더 형태로 제공.
- 별도 데스크톱·모바일 앱은 비대상.

## 6. 기능 요구사항 (MVP)

### 6-1. 프로젝트 / 템플릿 편집기

- 프로젝트 단위로 다수 사이즈 등록 (사용자 정의 폭·높이)
- 슬롯 기반 구조 (텍스트, 이미지, CTA, 배경 등 영역 지정)
- 재사용 컴포넌트 라이브러리 (버튼·로고·배지 등)
- 제한적 자유 레이아웃 (정해진 슬롯 외 영역에서 디자이너가 자유 배치)
- 사이즈별 개별 레이아웃 저장 → 자동 리사이즈 아님 (디자이너가 사이즈마다 직접 구성)

### 6-2. 데이터 / 소재 입력

- 텍스트·이미지 등 변수 데이터는 빌더 UI에서 직접 입력·업로드
- 이미지 소재는 자체 업로드 또는 AI 이미지 생성 연동으로 확보
- 외부 데이터 소스(Sheets, CMS, API)는 MVP 비대상

### 6-3. 생성 / 출력

- 트리거: 수동 (UI 버튼) → 사이즈별 일괄 렌더링
- 출력 포맷: PNG / JPG / WebP (우선), GIF (여유 시 추가)
- 출력물 다운로드 (일괄 zip 또는 개별)
- 외부 시스템 연동·자동 업로드는 비대상

### 6-4. 검수 / 승인 워크플로우

- 자동 검증
  - 오탈자 / 법적 표현 검증 (카피)
  - 이미지 적합성 (해상도·비율·안전 영역)
  - CTA 정상 작동
- 승인 단계 → 승인 없이는 다운로드·배포 불가
- 코멘트·리비전 가능

### 6-5. 프로젝트 / 히스토리 관리

- 프로젝트 버전 보존 (수정 이력)
- 이전 버전으로 롤백
- 프로젝트 링크 공유 (URL 기반, 보유자 접근)

## 7. Phase 2 범위

- 템플릿 변수 치환 자동화 (마케터 확장 전제 — 디자이너 템플릿에 데이터를 넣어 일괄 생성)
- AI 이미지 생성 연동 고도화
- AI 카피 생성 연동
- GIF / 간단한 애니메이션 출력

## 8. 기술 / 인프라

- 코드 저장: GitHub
- 배포: Vercel
- 클라이언트: 브라우저 기반 빌더 (Web)
- 렌더링 엔진은 별도 기술 검토 단계에서 결정 → Canvas/SVG vs Headless Chrome vs 외부 SaaS API
- 데이터 저장: 프로젝트·템플릿·소재·승인 이력 (스토리지 방식 추후 확정)
- API 제공: 내부용 다운로드만, 외부 공개 API 없음

## 9. 성능 / 운영 목표

- 규모: 팀/중규모 (하루 수천 건 이하)
- 동시 처리량·렌더링 SLA는 운영 시작 후 측정해서 조정
- 비용 한도 (AI 이미지 생성 API, 폰트 라이선스 등)는 Phase 2 진입 시 재산정

## 10. 보안 / 라이선스

- 이미지 저작권 관리
  - 업로드 이미지에 대한 출처·사용권 메타 보관
  - 라이선스 외 용도 사용 차단
- AI 생성물 관리
  - 생성 출처(모델·프롬프트) 메타 기록
  - 권리 표기 정책 명시
- 폰트는 빌더 내 등록된 라이선스 폰트만 사용

## 11. 성공 지표 (KPI)

- 배너 1세트 제작 소요 시간 (Before/After 비교)
- 디자이너 단위 시간당 처리 가능한 변형 수
- 반복 작업이 차지하는 비중 변화 (디자이너 자기 보고 + 작업 로그)
- 자동 검증 도입 후 재작업 비율 변화

## 12. 후속 결정 사항 (보류)

- 브랜드/디자인 토큰 시스템 도입 시점
- 외부 공개 시점 및 외부 사용자 권한 모델
- 광고 플랫폼 자동 업로드 도입 여부
- 실시간 다수 동시 편집 도입 여부
- 렌더링 엔진 최종 선택 (기술 검토 별도)
