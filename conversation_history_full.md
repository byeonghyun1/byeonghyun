**브랜드 이슈가** 중요한 포인트인 것 같습니다. 알바몬, 잡코리아, 워스피어 각각 스타일이 다르니까, 브랜드별로 에이전트를 분리할지 통합 구조로 갈지가 설계상 핵심 결정사항이고요.

## 어시스턴트 응답 #25

좋은 질문이에요. 하나씩 쉽게 설명할게요.

**1. 에러 기록 시스템**

지금은 에러가 나면 화면에 한 줄 뜨고 사라져요. 마치 메모 없이 전화 받는 것과 같아요. 해결 방법은 "업무 일지"를 만드는 거예요. 시스템이 뭔가를 할 때마다 자동으로 날짜, 시간, 어떤 작업이었는지, 성공했는지 실패했는지를 파일에 기록해두는 겁니다. 나중에 문제가 생기면 이 일지를 보고 "아, 어제 3시에 배너 만들다가 템플릿 연결에서 실패했구나" 하고 원인을 찾을 수 있어요.

**2. 설정 한 곳 관리**

지금은 API 키가 여기, 출력 경로가 저기, 폰트 설정이 또 다른 데 있어요. 비유하면 리모컨이 방마다 흩어져 있는 상태예요. 해결 방법은 설정 파일 하나를 만들어서 모든 설정을 한 곳에 모으는 거예요. "이 파일만 열면 시스템 전체 설정을 다 볼 수 있다"는 상태를 만드는 겁니다.

```
config.yaml (예시):
  api_key: "xxx..."
  output_folder: "./output"
  default_format: "png"
  default_brand: "albamon"
```

**3. 결과물 히스토리**

지금은 같은 걸 세 번 돌리면 파일 세 개가 쌓이는데, 어떤 조건으로 만든 건지 알 수 없어요. 해결 방법은 결과물을 만들 때마다 "이건 어떤 요청으로, 어떤 설정으로, 언제 만든 건지"를 함께 기록해두는 거예요. 디자인 작업할 때 파일명에 "v1", "v2" 붙이는 것과 비슷한데, 시스템이 자동으로 해주는 거죠. 이게 있어야 나중에 A/B 테스트("이 배너가 나은지 저 배너가 나은지")를 할 수 있어요.

**4. 이미지 에셋 처리**

카드뉴스는 텍스트 위주라 괜찮았지만, 배너나 포스터에는 상품 사진, 배경 이미지, 일러스트가 필요해요. 해결 방법은 이미지 창고를 만드는 거예요. 브랜드별로 자주 쓰는 이미지를 미리 정리해두고, 요청할 때 "이 이미지 써줘"라고 지정하거나, 나중에는 Midjourney나 Gemini 같은 AI 이미지 생성을 연결해서 자동으로 만드는 것도 가능합니다.

```
assets/
  albamon/
    backgrounds/    ← 배경 이미지 모음
    products/       ← 상품 사진
    illustrations/  ← 일러스트
  jobkorea/
    ...
```

**5. 출력 통로 분리**

지금은 파일 저장만 돼요. 근데 실무에서는 "만든 배너를 슬랙에 바로 보내줘", "구글 드라이브에 올려줘" 같은 상황이 생기잖아요. 해결 방법은 입력 쪽에 변환 플러그(어댑터)를 만든 것처럼, 출력 쪽에도 똑같이 만드는 거예요. 파일저장 플러그, 슬랙 전송 플러그, 드라이브 업로드 플러그 — 필요할 때 하나씩 추가하면 됩니다.

**6. 품질 피드백 루프**

이게 "시스템 자체 학습과 고도화" 목표의 핵심이에요. 결과물이 나오면 "이거 좋아 / 수정 필요 / 다시 만들어"라는 평가를 기록하는 거예요. 이 데이터가 쌓이면 "알바몬 브랜드에 활기찬 분위기로 요청하면 만족도가 높았다"같은 패턴을 알 수 있고, AI 카피 생성이나 스타일 선택을 점점 더 똑똑하게 만들 수 있습니다. 처음에는 간단하게 "좋아요/아쉬워요" 정도만 기록하고, 나중에 고도화하면 돼요.

1번(에러 기록)과 2번(설정 관리)은 시스템 만들면서 **같이 넣으면** 되고, 4번(이미지 에셋)은 배너·포스터 유형을 추가할 때, 5번(출력 통로)은 슬랙 연동할 때, 3번(히스토리)과 6번(피드백)은 시스템이 실무에 돌아간 후에 붙이면 됩니다.

### 2. 추가/수정한 파일

**신규 추가 (테스트 1):**
- `content_types/banner.py` — 배너의 copy_fields(title/subtitle/cta_text/bg_tagline)·sizes·프롬프트 선언

- `templates/card_news/portrait.html` — 1080×1350용 레이아웃

**수정 (엔진 3-축 지원):**
- `engine/schema.py` — `size_variant`, `copy_inputs` 2개 필드 추가

- `content_types/card_news.py` — `default_size: (1080,1080)` 튜플 → `sizes: {square, portrait}` dict 구조로 재편

### 3. 엔진 코드 실제 수정 분량

engine/ 폴더 내부 수정은 **약 40줄**입니다 (schema 6줄, generator의 _resolve_size 메서드 27줄 + 호출부 5줄, copy_writer 3줄). 이 수정은 테스트 1·2 양쪽 모두를 포괄하는 **일회성 구조 확장**이고, 이후에 배너·포스터·추가 사이즈가 늘어나도 engine/은 더 이상 건드릴 필요가 없습니다. 실제로 배너 추가는 engine 수정 0줄로 끝났고, portrait 사이즈 추가도 content_types/card_news.py 딕셔너리 한 항목과 템플릿 하나만으로 끝났습니다.

### 4. 3축 분리 상태 평가

**형식 축 (content_type)** — ⭐⭐⭐⭐⭐ 잘 분리됨. engine은 "card_news"라는 단어를 모르고, content_types 레지스트리의 copy_fields·sizes·pillow_renderer를 동적으로 읽습니다. 카드뉴스와 배너가 서로 다른 필드 집합(badge/description vs subtitle/cta_text/bg_tagline)을 가져도 copy_writer가 metadata-driven으로 처리합니다.

**스타일 축 (brand)** — ⭐⭐⭐⭐ 잘 분리됨. brands/ 패키지가 content_types/와 병렬로 존재하고, `resolve_brand_for_content()`가 `overrides[content_type]`를 deep-merge해서 "알바몬의 배너 전용 배경색"같은 예외도 선언적으로 처리됩니다. 엔진이 브랜드 이름을 하드코딩하지 않습니다.

**규격 축 (size)** — ⭐⭐⭐⭐ 구조적으로 분리됨. content_type 안에 sizes dict로 중첩된 것은 자연스러운 설계(배너와 카드뉴스의 사이즈 목록이 서로 다르므로). `size_variant`로 선택만 하면 템플릿도 자동으로 따라갑니다.

### 5. 남아 있는 구조적 위험 요소

1. **템플릿-카피 필드 계약이 암묵적**  
   copy_fields에 필드를 선언해도 템플릿에서 `{{ subtitle }}`을 실제로 써주지 않으면 조용히 누락됩니다. 지금은 배너 4개 필드 × 카드뉴스 4개 필드 수준이라 괜찮지만, 상세페이지처럼 10개 이상 필드가 나오면 "선언했는데 렌더링 안 됨" 버그가 쉽게 생길 수 있습니다. → 추후 템플릿이 기대하는 변수 목록을 content_type 설정에 명시하고, 렌더링 전에 검증하는 단계가 있으면 좋습니다.

2. **Pillow 폴백은 콘텐츠 유형마다 직접 만들어야 함**  
   배너에는 아직 Pillow 폴백이 없습니다. Playwright 환경이 깨진 사용자는 배너를 못 만듭니다. 레지스트리의 `pillow_renderer: None`인 유형은 사전 경고를 띄워주는 게 안전합니다.

3. **sizes dict와 구형 튜플 구조가 둘 다 허용됨**  
   `_resolve_size`가 하위 호환으로 둘 다 받아주는데, 장기적으로는 모든 content_type을 sizes dict로 통일하고 튜플 경로를 제거하는 쪽이 유지보수에 유리합니다.

4. **브랜드 × 사이즈 override는 아직 없음**  
   현재 overrides는 content_type 단위까지만 허용됩니다. "알바몬의 카드뉴스 portrait은 다른 로고를 써야 함"같은 요구가 생기면 구조 확장이 한 번 더 필요합니다. 당장은 불필요하지만 언젠가 나올 수 있는 요청입니다.

5. **copy_inputs 키는 자유 문자열**  
   배너의 `cta_text`, `bg_tagline`처럼 유형 고유 필드는 copy_inputs dict에 넣는데, 오타가 있어도 조용히 빈 값으로 처리됩니다. copy_fields에 선언된 키인지 검증하는 경고가 있으면 디자이너가 실수하기 어려워집니다.

전체적으로 "지금 돌아가게 만드는" 임시 방식이 아니라, **새 콘텐츠 유형/브랜드/사이즈 추가 시 엔진을 건드리지 않아도 되는 방향**으로 3축이 깔렸습니다. 다음 확장(포스터·상세페이지·새 브랜드)은 engine/ 폴더를 안 건드리고 선언 파일과 템플릿만 추가하면 될 것으로 보입니다.

## 1. 수정된 입력 구조

`DesignRequest`의 3축 필드입니다. 사용자는 이 세 값을 **입력 단계에서 각각 선택**하고, 내부에서는 하나의 `DesignRequest` 객체로 함께 흐릅니다.

```python
@dataclass
class DesignRequest:
    # ── 사용자 입력 (공통) ──
    topic: str = ""
    target: str = ""
    purpose: str = ""
    mood: str = ""
    copy_mode: str = "manual"
    cta: str = ""

    # ── 3축 (형식 × 규격 × 스타일) ──
    content_type: str = "card_news"   # card_news | banner | poster | detail_page ...
    size: str = ""                    # 1080x1080 | 1080x1350 | 1200x628 | A4_vertical ...
    brand: str = ""                   # albamon | jobkorea | worxphere ...

    # ── 유형별 고유 카피 (subtitle, cta_text, date_range 등) ──
    copy_inputs: dict = field(default_factory=dict)

    # ── 공통 카피 슬롯 ──
    title / description / badge / footer
    ...
```

CLI도 세 축을 노출합니다:
```
python main.py "신규 가입 이벤트" --type card_news --size 1080x1350 --brand jobkorea
python main.py "알바 페스티벌"    --type poster    --size A4_vertical  --brand albamon
```

## 2. 전체 흐름에서 세 값이 어떻게 사용되는지

```
[입력]
  DesignRequest(content_type, size, brand, ...)
        │
        ▼
[엔진 공통 파이프라인 — 세 축 이름을 하드코딩하지 않음]
  (1) content_types.get_config(content_type)      ← 형식 축 → copy_fields, sizes, pillow_renderer
  (2) engine._resolve_size(request, config)       ← 규격 축 → width/height/template 선택
  (3) brands.resolve_brand_for_content(brand, ct) ← 스타일 축 → colors, fonts, overrides
  (4) copy_writer.generate_context(request)       ← 위 3개를 병합해 템플릿 변수 dict 생성
  (5) renderer.render(...)                        ← Playwright → Pillow 폴백 → HTML
        │
        ▼
[출력]
  DesignResult(output_path, html_path, renderer_used, context, ...)
```

각 축의 역할:
- **content_type → 형태/구조**: 어떤 카피 필드를 받고(badge/subtitle/date_range 등), 어떤 Pillow 폴백 레이아웃을 쓰는지 결정
- **size → 규격/레이아웃**: 너비·높이뿐 아니라 **해당 사이즈 전용 템플릿**을 선택. 같은 card_news라도 1080x1080은 basic.html, 1080x1350은 portrait.html로 분기
- **brand → 스타일**: 색상 6슬롯·폰트·CSS 변수만 주입하고 구조에는 관여하지 않음. 필요 시 `overrides[content_type]`로 특정 유형에만 다른 스타일도 가능

검증 결과 (4/4 성공):
| 조합 | 실제 사이즈 | 결과 |

## 3. 새 콘텐츠 유형 추가 시 바뀌는 파일

**엔진은 건드리지 않습니다.** 아래 3가지만 합니다.

1. `content_types/new_type.py` — 설정 dict 작성 (copy_fields, sizes, default_size, (옵션)pillow_renderer, ai_prompt)
2. `templates/new_type/xxx.html` — 1개 이상의 레이아웃 템플릿
3. `content_types/__init__.py`의 `REGISTRY`에 **한 줄 등록**

포스터가 실제로 이 규칙대로만 추가됐습니다. `engine/` 폴더는 수정 0줄.

## 4. 새 사이즈 추가 시 바뀌는 파일

**엔진은 건드리지 않습니다.** 해당 콘텐츠 유형의 설정 파일 `sizes` dict에 항목 하나 추가만 하면 됩니다.

```python
# content_types/poster.py
"sizes": {
    "A4_vertical":       { "width": 595,  "height": 842,  "template": "poster/a4.html" },
    "A4_vertical_print": { "width": 2480, "height": 3508, "template": "poster/a4.html" },
    "A3_vertical":       { "width": 842,  "height": 1191, "template": "poster/a3.html" }, # ← 추가
},
```

같은 템플릿을 재사용하면 HTML 추가도 불필요. 다른 레이아웃을 쓰고 싶으면 `templates/poster/a3.html` 하나만 추가합니다.

사이즈는 **단순 옵션이 아니라 템플릿과 묶인 핵심 값**입니다. 엔진의 `_resolve_size()`가 `sizes[size].template`을 읽어 해당 사이즈 전용 레이아웃으로 분기합니다.

## 5. 새 브랜드 추가 시 바뀌는 파일

**엔진과 content_types 양쪽 모두 건드리지 않습니다.**

1. `brands/new_brand.py` — `name`, `colors`(6슬롯), `typography`, (옵션)`overrides[content_type]` 선언
2. `brands/__init__.py`의 `REGISTRY`에 **한 줄 등록**

## 6. 코드가 꼬이지 않도록 적용한 설계 원칙

**① 축 이름을 엔진 코드에 하드코딩하지 않음**
`card_news`, `albamon`, `1080x1080` 같은 문자열이 engine/ 어디에도 if/elif로 등장하지 않습니다. 모두 레지스트리 dict 조회로 처리됩니다.

**② 레지스트리 + 설정 파일 패턴 (Open/Closed)**
`content_types/`, `brands/` 두 패키지는 "열려 있음"(새 파일 추가 자유), engine/은 "닫혀 있음"(수정 불필요). 새 유형·브랜드는 파일 추가 + 레지스트리 한 줄 등록으로 끝.

**③ 3축 직교성**
형식 축(content_type), 규격 축(size), 스타일 축(brand)은 각각 별도 패키지/필드로 분리. 카드뉴스 × 1080x1350 × 잡코리아처럼 임의 조합이 자동으로 됩니다.

**④ 규격 축은 유형 내부에 nest**
사이즈는 콘텐츠 유형마다 의미가 다르므로(배너의 1200x628과 포스터의 A4는 섞이면 안 됨), 각 content_type 설정의 `sizes` dict 안에 둡니다. 허용 사이즈 목록을 유형별로 관리하라는 추가 요구사항 그대로입니다.

**⑤ 브랜드 × 유형 조합은 override로 선언**
`brand.overrides[content_type]`로 "이 브랜드의 이 유형에서만 다른 색"을 선언적으로 표현. 엔진에 분기문이 생기지 않습니다.

**⑥ 카피 필드도 설정이 자기 자신을 선언**
카드뉴스는 badge/title/description/footer, 배너는 title/subtitle/cta_text/bg_tagline, 포스터는 title/subtitle/date_range/place/footer. copy_writer는 이 메타데이터(`required`, `fallback_field`, `auto_extract_from`, `keywords`, `fallback`)를 읽어 동작하므로, 유형이 늘어도 copy_writer를 건드릴 일이 없습니다.

**⑦ 템플릿은 변수 치환만**
Jinja2 템플릿에는 `{{ bg_color }}` 같은 **구조 변수만** 있고 브랜드 이름이나 사이즈 이름이 박히지 않습니다.

**⑧ `copy_inputs` 확장 포트**
공통 슬롯에 없는 유형 고유 카피(cta_text, date_range 등)는 `copy_inputs` dict로 전달. 새 필드가 생겨도 DesignRequest의 시그니처는 바뀌지 않습니다.

**⑨ 하위 호환 경로 유지**
`_resolve_size`가 신형 `sizes` dict와 구형 `default_size` 튜플 둘 다 처리해서, 기존 코드/데이터를 깨지 않고 점진 마이그레이션이 가능합니다.

### B 방식(Figma 플러그인)으로 확장하는 로드맵

지금 구조에서 B로 넘어갈 때 필요한 작업은 3가지뿐입니다:

1. `engine/figma_json_exporter.py` 신규 작성 — `Frame → Figma 플러그인이 이해하는 JSON 트리` 변환. LayerTree의 같은 5개 타입을 Figma API의 `figma.createFrame() / createText() / createRectangle() / createEllipse()` 호출로 매핑하면 됩니다.
2. `generator.py`의 `_render_layer_output()`에서 `output_format == "figma_json"` 분기에 `NotImplementedError` 대신 방금 만든 exporter 호출로 교체 (한 줄).
3. Figma 플러그인 쪽: 사용자가 `.figma.json`을 드롭/업로드하면 JSON을 파싱해서 위 API로 노드 생성. 플러그인 자체는 50~100줄짜리 TypeScript로 끝납니다.

**중요한 건 `content_types/*_layers.py`는 B로 가도 수정하지 않는다**는 점입니다. 레이아웃 정의는 한 곳(레이어 빌더)에만 있고, 출력 포맷은 계속 늘려나갈 수 있습니다.

### 남아 있는 구조적 위험 / 개선 여지

1. **텍스트 줄바꿈은 근사치**: SVG 단계에서 폭 기준으로 단순 계산. Figma에서는 각 Text 레이어가 "auto-width"로 들어와 줄바꿈이 달라질 수 있음. 필요하면 Text 레이어에 명시적 width(고정 폭 박스)를 주도록 exporter 옵션 추가 가능.
2. **배너·포스터는 아직 layer_builder 없음**: 지금은 카드뉴스만 SVG 출력이 됩니다. 배너/포스터도 `*_layers.py` 하나씩 추가하면 동일 구조로 동작. layer_builder가 없는 유형에 `--format svg`를 주면 "layer_builder가 없어 지원하지 않음"이라는 명시적 에러가 나옵니다.
3. **이미지 레이어는 선언만 있고 실사용 전**: 추후 배너/포스터에서 사진 배경이 들어가면 `ImageLayer`를 쓰고, SVG에서는 `<image href="...">`, Figma JSON에서는 이미지 해시로 각각 매핑하면 됩니다.

## 3. 더미 카피 생성 방식

`engine/dummy_copy.py`에 두 함수가 있습니다.

**generate_dummy** — copy_fields를 한 필드씩 돌면서 다음 우선순위로 채웁니다:
1. 사용자가 직접 입력한 값(`request.title`, `request.copy_inputs[...]`)이 있으면 그대로 사용
2. 필드 메타에 `fallback` 고정 문자열이 있으면 그것
3. 필드 이름 패턴(`title`, `subtitle`, `cta_text`, `bg_tagline`, `date_range`, `place`, `footer`, `badge`)에 따라 정의된 규칙으로 생성. 예: `title`은 분위기 형용사 + topic 결합("프리미엄 대기업 신입 채용 오픈"), `cta_text`는 `purpose`별 사전(`signup → 무료 가입하기`)
4. 마지막으로 `max_length`로 자동 절단

**refine_dummy** — 사용자 입력을 가볍게 다듬는 흉내. 제목엔 분위기 형용사를 앞에 붙이고("AI로 운영을 자동화하세요" → "프리미엄 AI로 운영을 자동화하세요"), 설명엔 끝에 클로저 한 마디 덧붙입니다.

규칙은 **필드 이름과 메타 데이터만** 보고 동작하므로, 새 콘텐츠 유형이 같은 명명 규칙(`title/subtitle/cta_text/...`)을 쓰면 더미도 자동으로 적절한 값을 채웁니다. 실제 Anthropic API 붙일 때는 이 폴백이 그대로 안전망 역할을 합니다 (`api_key`가 있는데 호출이 실패하면 더미로 떨어짐).

## 4. SVG 생성 방식

핵심은 **HTML을 파싱하지 않고**, 콘텐츠 유형마다 선언된 `layer_builder` 함수가 **중간 표현 LayerTree**를 직접 만든다는 점입니다.

```
context (카피 + 색상)  ──┐
                          ├─→ layer_builder(context, w, h)  →  Frame {Text, Rect, Ellipse, ...}
(width, height)        ──┘                                              │
                                                                         ▼
                                                   svg_exporter.export_svg(frame)
                                                   → <svg> 안에 <text>·<rect>·<ellipse>
```

- 텍스트는 `<text>` + `<tspan>`으로 직렬화되어 Figma에서 **편집 가능한 Text 레이어**로 들어옵니다
- 모든 노드에 `data-figma-name` 속성을 붙여 Figma의 레이어 패널에 의미 있는 이름이 보이게 함 (`Title`, `Badge/Background`, `Decoration/Top-Right` 등)
- `<defs>` 안에 Google Fonts(Noto Sans KR) 임포트와 `Apple SD Gothic Neo / Malgun Gothic` 폴백 스택을 넣어 한글 깨짐 방지
- 텍스트 줄바꿈은 `_wrap_text()`가 폭 기준으로 근사해서 `<tspan>`으로 분리

같은 LayerTree가 나중에 B 방식(Figma 플러그인 JSON)으로도 갈 수 있도록, **layer_builder 코드는 출력 포맷을 모릅니다**.

## 5. Figma에서 테스트하는 방법

1. `output/prototype/` 폴더에서 원하는 `.svg` 파일을 선택
2. Figma 파일을 열고 캔버스에 **드래그&드롭** (또는 `File → Place image → SVG 선택`)
3. 들어온 그룹을 **우클릭 → Ungroup** 하면 레이어 패널에 다음과 같이 나타납니다:
   ```
   Background
   Decoration/Top-Right
   Decoration/Bottom-Left
   Badge/Background
   Badge/Text          ← 더블클릭으로 텍스트 수정
   Title               ← 폰트/크기/색 변경 가능
   Divider
   Description
   Footer
   ```
4. 텍스트 레이어를 더블클릭해서 문구 교체, 도형은 채움색을 바로 변경
5. 수정 후 `Export` 패널에서 PNG/JPG/PDF로 내보내기

확인 포인트 3가지만 봐주시면 됩니다: (1) 텍스트가 진짜 Text 레이어로 들어왔는지 (2) 한글 폰트가 정상 표시되는지 (3) 원/사각형 도형이 벡터로 분리됐는지.
